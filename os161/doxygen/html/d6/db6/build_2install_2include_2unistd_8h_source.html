50/*
51 * Prototypes for OS/161 system calls.
52 *
53 * Note that the following system calls are prototyped in other
54 * header files, as follows:
55 *
56 * stat: sys/stat.h
57 * fstat: sys/stat.h
58 * lstat: sys/stat.h
59 * mkdir: sys/stat.h
60 *
61 * If this were standard Unix, more prototypes would go in other
62 * header files as well, as follows:
63 *
64 * waitpid: sys/wait.h
65 * open: fcntl.h or sys/fcntl.h
66 * reboot: sys/reboot.h
67 * ioctl: sys/ioctl.h
68 * remove: stdio.h
69 * rename: stdio.h
70 * time: time.h
71 *
72 * Also note that the prototypes for open() and mkdir() contain, for
73 * compatibility with Unix, an extra argument that is not meaningful
74 * in OS/161. This is the "mode" (file permissions) for a newly created
75 * object. (With open, if no file is created, this is ignored, and the
76 * call prototype is gimmicked so it doesn't have to be passed either.)
77 *
78 * You should ignore these arguments in the OS/161 kernel unless you're
79 * implementing security and file permissions.
80 *
81 * If you are implementing security and file permissions and using a
82 * model different from Unix so that you need different arguments to
83 * these calls, you may make appropriate changes, or define new syscalls
84 * with different names and take the old ones out, or whatever.
85 *
86 * As a general rule of thumb, however, while you can make as many new
87 * syscalls of your own as you like, you shouldn't change the
88 * definitions of the ones that are already here. They've been written
89 * to be pretty much compatible with Unix, and the teaching staff has
90 * test code that expects them to behave in particular ways.
91 *
92 * Of course, if you want to redesign the user/kernel API and make a
93 * lot of work for yourself, feel free, just contact the teaching
94 * staff beforehand. :-)
95 *
96 * The categories (required/recommended/optional) are guesses - check
97 * the text of the various assignments for an authoritative list.
98 */
99
